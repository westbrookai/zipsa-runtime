# Docker Compose Agent Deployment Design

**Date:** 2026-04-24
**Status:** Approved
**Phase:** 1 (Script-based), with Phase 2 (Web UI) migration path

---

## Executive Summary

This design introduces Docker Compose-based deployment for domain-specific agents (e.g., Xero Agent, Sales Agent) built on zipsa-runtime. The goal is to eliminate technical barriers for non-technical customers by replacing manual `docker run` commands with simple setup and start scripts.

**Key Goals:**
- Enable customers with zero Docker knowledge to run SKILL-based agents
- Provide interactive setup for API keys and runtime selection
- Support multiple domain agents with isolated configurations
- Create a foundation for Phase 2 Web UI migration

### Design Principles

1. **Single Compose File** - One `docker-compose.yml` per domain agent, runtime selected via environment variable
2. **Interactive Setup** - `setup.sh` guides users through initial configuration (API keys, runtime selection)
3. **Simple Execution** - `start.sh` and `stop.sh` are thin wrappers around `docker compose` commands
4. **Phase 2 Ready** - Web UI can reuse the same `docker-compose.yml` structure without breaking changes

## File Structure

### Domain Agent Repository Layout

```
xero-agent/                          # Example domain agent repo
├── docker-compose.yml               # Container orchestration
├── .env.example                     # Template for required variables
├── .env                             # Generated by setup.sh (gitignored)
├── .gitignore                       # Excludes .env and logs
├── setup.sh                         # Interactive first-time configuration
├── setup.bat                        # Windows equivalent
├── start.sh                         # Start the agent container
├── start.bat                        # Windows equivalent
├── stop.sh                          # Stop the agent container
├── stop.bat                         # Windows equivalent
├── README.md                        # Customer-facing documentation
├── skills/                          # Domain-specific SKILL definitions
│   └── xero-accounting/
│       ├── skill.md                 # SKILL implementation
│       └── metadata.json            # SKILL metadata
├── servers.json                     # MCP server configuration
└── workspace/                       # User's working directory (created on first run)
```

### File Responsibilities

#### docker-compose.yml
- Defines single service named after the domain (e.g., `xero-agent`)
- Pulls `ghcr.io/westbrookai/zipsa-runtime:latest`
- Mounts skills and servers.json into container
- Reads runtime selection from `.env`
- Sets working directory to `/workspace`
- Enables interactive terminal (stdin_open, tty)

#### .env.example
Template file showing required environment variables:
```bash
# API Keys
ANTHROPIC_API_KEY=your-key-here
GOOGLE_OAUTH_CLIENT_ID=your-id-here
GOOGLE_OAUTH_CLIENT_SECRET=your-secret-here

# Runtime Selection (claude, codex, or gemini)
RUNTIME_AGENT=claude
```

#### setup.sh
- Displays welcome message
- Checks Docker installation and availability
- Prompts for API keys (hidden input for security)
- Validates API key format (basic check)
- Prompts for runtime selection (Claude/Codex/Gemini)
- Generates `.env` from user input
- Pulls latest zipsa-runtime image
- Shows success message with next steps

#### start.sh
- Checks `.env` exists (prompts to run setup.sh if missing)
- Creates workspace directory if needed
- Executes `docker compose up -d` or `docker compose up` based on flag
- Shows logs and connection instructions
- Attaches to container logs

#### stop.sh
- Executes `docker compose down`
- Optionally preserves or clears volumes
- Displays confirmation message

## Docker Compose Configuration

### docker-compose.yml Structure

```yaml
version: '3.8'

services:
  xero-agent:
    image: ghcr.io/westbrookai/zipsa-runtime:latest
    container_name: xero-agent
    stdin_open: true
    tty: true

    environment:
      # API Keys from .env
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
      - GOOGLE_OAUTH_CLIENT_ID=${GOOGLE_OAUTH_CLIENT_ID:-}
      - GOOGLE_OAUTH_CLIENT_SECRET=${GOOGLE_OAUTH_CLIENT_SECRET:-}

      # Runtime selection
      - RUNTIME_AGENT=${RUNTIME_AGENT:-claude}

      # Terminal settings
      - TERM=xterm-256color

    volumes:
      # Mount skills directory (read-only for safety)
      - ./skills:/root/.claude/skills:ro

      # Mount MCP server config
      - ./servers.json:/app/servers.json:ro

      # Mount workspace for user projects
      - ./workspace:/workspace

      # Optional: persist Claude settings across restarts
      - claude-config:/root/.claude/config

    working_dir: /workspace

    # Command varies by runtime
    command: >
      sh -c "
        case ${RUNTIME_AGENT} in
          claude) claude --mcp-config /app/servers.json ;;
          codex) codex --mcp-config /app/servers.json ;;
          gemini) gemini --mcp-config /app/servers.json ;;
          *) echo 'Invalid RUNTIME_AGENT' && exit 1 ;;
        esac
      "

    # Restart policy
    restart: unless-stopped

volumes:
  claude-config:
    name: xero-agent-config
```

### Configuration Decisions

**Interactive Terminal:**
- `stdin_open: true` + `tty: true` enables interactive agent conversation
- Required for chat-based agent interaction

**Volume Mounts:**
- **skills/** (read-only) - Prevents accidental modification by container
- **servers.json** (read-only) - MCP configuration protection
- **workspace/** (read-write) - Customer's working directory for projects
- **claude-config** (named volume) - Persists agent settings across restarts

**Runtime Selection:**
- Single service definition with shell case statement
- Environment variable `RUNTIME_AGENT` determines which agent binary to execute
- Defaults to `claude` if not specified

**Restart Policy:**
- `unless-stopped` - Container auto-restarts unless explicitly stopped
- Survives system reboots and Docker daemon restarts

## Script Implementation Details

### setup.sh Pseudocode

```bash
#!/bin/bash
set -e  # Exit on error

# 1. Welcome message
echo "Welcome to [Domain Agent] Setup"

# 2. Check Docker availability
if ! command -v docker &> /dev/null; then
    echo "Error: Docker is not installed"
    echo "Please install Docker Desktop from https://docker.com"
    exit 1
fi

# Verify Docker daemon is running
if ! docker info &> /dev/null; then
    echo "Error: Docker is not running"
    echo "Please start Docker Desktop and try again"
    exit 1
fi

# 3. Check for existing .env
if [ -f .env ]; then
    read -p ".env already exists. Reconfigure? (y/n): " reconfigure
    if [ "$reconfigure" != "y" ]; then
        echo "Keeping existing configuration"
        exit 0
    fi
    echo "Reconfiguring..."
fi

# 4. Prompt for API keys
echo ""
read -sp "Enter your ANTHROPIC_API_KEY: " api_key
echo ""

# Basic validation (starts with sk-)
if [[ ! $api_key =~ ^sk- ]]; then
    echo "⚠ Warning: API key should start with 'sk-'"
    read -p "Continue anyway? (y/n): " continue
    if [ "$continue" != "y" ]; then
        echo "Setup cancelled"
        exit 1
    fi
fi

# 5. Optional: Additional API keys
echo ""
read -p "Do you need Google OAuth credentials? (y/n): " needs_google
google_id=""
google_secret=""

if [ "$needs_google" = "y" ]; then
    read -p "GOOGLE_OAUTH_CLIENT_ID: " google_id
    read -sp "GOOGLE_OAUTH_CLIENT_SECRET: " google_secret
    echo ""
fi

# 6. Runtime selection
echo ""
echo "Select runtime agent:"
echo "1) Claude (Recommended)"
echo "2) Codex"
echo "3) Gemini"
read -p "Choice (1-3): " runtime_choice

case $runtime_choice in
    1) runtime="claude" ;;
    2) runtime="codex" ;;
    3) runtime="gemini" ;;
    *)
        echo "Invalid choice. Defaulting to Claude."
        runtime="claude"
        ;;
esac

# 7. Generate .env file
cat > .env << EOF
# Generated by setup.sh on $(date)
ANTHROPIC_API_KEY=${api_key}
RUNTIME_AGENT=${runtime}
EOF

if [ "$needs_google" = "y" ]; then
    cat >> .env << EOF
GOOGLE_OAUTH_CLIENT_ID=${google_id}
GOOGLE_OAUTH_CLIENT_SECRET=${google_secret}
EOF
fi

echo "✓ Configuration saved to .env"

# 8. Pull latest runtime image
echo ""
echo "Pulling latest zipsa-runtime image..."
if docker pull ghcr.io/westbrookai/zipsa-runtime:latest; then
    echo "✓ Runtime image downloaded"
else
    echo "⚠ Failed to pull image. Will retry on first start."
fi

# 9. Success message
echo ""
echo "════════════════════════════════════════"
echo "✓ Setup complete!"
echo "════════════════════════════════════════"
echo ""
echo "Next steps:"
echo "  1. Run './start.sh' to launch the agent"
echo "  2. Wait for initialization to complete"
echo "  3. Start chatting with your agent"
echo ""
```

### start.sh Pseudocode

```bash
#!/bin/bash
set -e

# 1. Check .env exists
if [ ! -f .env ]; then
    echo "Error: Configuration not found"
    echo "Please run './setup.sh' first"
    exit 1
fi

# 2. Load environment variables
source .env

# 3. Check workspace directory exists
if [ ! -d workspace ]; then
    echo "Creating workspace directory..."
    mkdir -p workspace
fi

# 4. Start container
echo "Starting ${RUNTIME_AGENT} agent..."
docker compose up -d

# Wait for container to be healthy
echo "Waiting for container to initialize..."
sleep 2

# 5. Check if container is running
if docker compose ps | grep -q "running"; then
    echo "✓ Agent started successfully"
else
    echo "✗ Agent failed to start"
    echo "Last 20 lines of logs:"
    docker compose logs --tail=20
    exit 1
fi

# 6. Show logs
echo ""
echo "════════════════════════════════════════"
echo "Agent is running!"
echo "════════════════════════════════════════"
echo ""
echo "Attaching to logs (Ctrl+C to detach, won't stop container)..."
echo ""
docker compose logs -f
```

### stop.sh Pseudocode

```bash
#!/bin/bash

# 1. Check if container is running
if ! docker compose ps | grep -q "running"; then
    echo "Agent is not running"
    exit 0
fi

# 2. Stop and remove containers
echo "Stopping agent..."
docker compose down

# Note: Volumes are preserved by default
# To also remove volumes: docker compose down -v

echo "✓ Agent stopped"
echo ""
echo "Your workspace and configuration are preserved"
echo "Run './start.sh' to restart the agent"
```

## Error Handling & Edge Cases

### Common Failure Scenarios

#### 1. Docker Not Installed/Running

**Detection:**
```bash
# In setup.sh and start.sh
if ! command -v docker &> /dev/null; then
    # Docker not installed
fi

if ! docker info &> /dev/null; then
    # Docker daemon not running
fi
```

**Error Messages:**
```
Error: Docker is not installed
Please install Docker Desktop from https://docker.com
  - macOS: https://docs.docker.com/desktop/install/mac-install/
  - Windows: https://docs.docker.com/desktop/install/windows-install/
  - Linux: https://docs.docker.com/desktop/install/linux-install/
```

```
Error: Docker is not running
Please start Docker Desktop and try again
```

#### 2. Missing .env File

**Detection:**
```bash
# In start.sh
if [ ! -f .env ]; then
    # Configuration missing
fi
```

**Error Message:**
```
Error: Configuration not found
Please run './setup.sh' first to configure your agent
```

#### 3. Invalid API Key Format

**Detection:**
```bash
# In setup.sh
if [[ ! $api_key =~ ^sk- ]]; then
    # Warning for invalid format
fi
```

**Warning Message:**
```
⚠ Warning: API key should start with 'sk-'
The format looks incorrect. Double-check your key.
Continue anyway? (y/n):
```

**Note:** We don't test API keys by making actual calls (container handles that), only basic format validation.

#### 4. Container Fails to Start

**Detection:**
```bash
# In start.sh
docker compose up -d
if ! docker compose ps | grep -q "running"; then
    # Container not running
fi
```

**Error Message:**
```
✗ Agent failed to start

Last 20 lines of logs:
[docker compose logs --tail=20 output]

Common issues:
  - Invalid API key (check .env file)
  - Network connectivity (check internet connection)
  - Port conflicts (stop other containers)

To reconfigure: run './setup.sh'
To see full logs: docker compose logs
```

#### 5. Port Conflicts (Future)

**Scenario:** If future versions expose ports and conflict occurs

**Detection:**
Docker Compose will report port binding errors

**Error Message:**
```
Error: Port already in use
Another service is using the required port.

To find the conflicting process:
  - macOS/Linux: lsof -i :PORT_NUMBER
  - Windows: netstat -ano | findstr :PORT_NUMBER

Stop the conflicting service or change the port in docker-compose.yml
```

#### 6. Permission Issues (Linux)

**Detection:**
```
Error: permission denied while trying to connect to Docker daemon socket
```

**Error Message:**
```
Error: Docker permission denied

Your user doesn't have permission to use Docker.

Fix:
  1. sudo usermod -aG docker $USER
  2. Log out and log back in
  3. Run './setup.sh' again

Alternative: Run with sudo (not recommended)
```

#### 7. Network Issues During Image Pull

**Detection:**
```bash
# In setup.sh
if ! docker pull ghcr.io/westbrookai/zipsa-runtime:latest; then
    # Pull failed
fi
```

**Error Message:**
```
⚠ Failed to pull image. Will retry on first start.

If this persists:
  - Check internet connection
  - Verify you can access ghcr.io
  - Try again later

The agent will attempt to pull on first start.
```

#### 8. Workspace Permission Issues

**Detection:**
Container cannot write to workspace directory

**Error Message:**
```
Warning: Workspace directory has restricted permissions

Fix:
  chmod 755 workspace

Or run with your user ID:
  Edit docker-compose.yml and add:
    user: "$(id -u):$(id -g)"
```

### Recovery Strategies

**General Principles:**
- All scripts use `set -e` for fail-fast behavior
- Clear error messages with actionable next steps
- Preserve existing `.env` during failed setup
- Logs accessible via `docker compose logs`
- Each error suggests specific remediation

**Cleanup Commands:**
```bash
# Full reset
./stop.sh
rm .env
docker compose down -v

# Then run setup again
./setup.sh
```

## Testing Strategy

### Integration Testing

Each domain agent repository should include `test-setup.sh`:

```bash
#!/bin/bash
# Integration tests for domain agent setup

set -e

echo "Running integration tests..."

# Test 1: setup.sh creates valid .env
test_setup_generates_env() {
    echo "Test 1: setup.sh generates .env"

    # Cleanup
    rm -f .env

    # Simulate user input (api key, no google, claude)
    echo -e "sk-test-key\nn\n1\n" | ./setup.sh > /dev/null

    if [ -f .env ]; then
        echo "  ✓ .env created"
    else
        echo "  ✗ .env not created"
        exit 1
    fi

    # Verify contents
    grep -q "ANTHROPIC_API_KEY=sk-test-key" .env || exit 1
    grep -q "RUNTIME_AGENT=claude" .env || exit 1
    echo "  ✓ .env has correct content"
}

# Test 2: docker-compose.yml is valid
test_compose_valid() {
    echo "Test 2: docker-compose.yml validation"

    if docker compose config > /dev/null 2>&1; then
        echo "  ✓ docker-compose.yml is valid"
    else
        echo "  ✗ docker-compose.yml has syntax errors"
        docker compose config
        exit 1
    fi
}

# Test 3: start.sh requires .env
test_start_requires_env() {
    echo "Test 3: start.sh requires .env"

    rm -f .env

    if ./start.sh 2>&1 | grep -q "Please run './setup.sh' first"; then
        echo "  ✓ start.sh properly checks for .env"
    else
        echo "  ✗ start.sh missing .env check"
        exit 1
    fi
}

# Test 4: Skills directory exists
test_skills_exist() {
    echo "Test 4: Skills directory structure"

    if [ -d skills ]; then
        echo "  ✓ skills/ directory exists"
    else
        echo "  ✗ skills/ directory missing"
        exit 1
    fi
}

# Test 5: MCP config exists
test_mcp_config_exists() {
    echo "Test 5: MCP configuration"

    if [ -f servers.json ]; then
        echo "  ✓ servers.json exists"
    else
        echo "  ✗ servers.json missing"
        exit 1
    fi

    # Validate JSON
    if cat servers.json | python3 -m json.tool > /dev/null 2>&1; then
        echo "  ✓ servers.json is valid JSON"
    else
        echo "  ✗ servers.json is invalid JSON"
        exit 1
    fi
}

# Test 6: .gitignore includes .env
test_gitignore() {
    echo "Test 6: .gitignore configuration"

    if grep -q "^\.env$" .gitignore; then
        echo "  ✓ .env is gitignored"
    else
        echo "  ✗ .env not in .gitignore"
        exit 1
    fi
}

# Run all tests
test_setup_generates_env
test_compose_valid
test_start_requires_env
test_skills_exist
test_mcp_config_exists
test_gitignore

echo ""
echo "════════════════════════════════════════"
echo "All tests passed!"
echo "════════════════════════════════════════"
```

## Phase 2 Migration Path (Web UI)

### Design Compatibility

Phase 1 is intentionally structured for seamless Phase 2 migration:

**Reusable Components:**
- `docker-compose.yml` remains unchanged (Web UI orchestrates it)
- `.env` file format stays the same (Web UI writes to it)
- Volume mounts and network config preserved
- Skills and `servers.json` structure unchanged

### Phase 2 Architecture

```
┌──────────────────────────────────────────┐
│         Web UI (New in Phase 2)          │
│  ┌────────────────────────────────────┐  │
│  │  Backend API Server                │  │
│  │  - Express/FastAPI                 │  │
│  │  - Manages docker compose commands │  │
│  │  - Stores configurations           │  │
│  │  - WebSocket for logs              │  │
│  └────────────────────────────────────┘  │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │  Frontend Dashboard                │  │
│  │  - Agent selection UI              │  │
│  │  - Status monitoring               │  │
│  │  - Log viewer                      │  │
│  │  - Configuration manager           │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
                    │
                    ▼
         Reuses existing infrastructure:
         - docker-compose.yml (same files)
         - .env (same format)
         - skills/ (same location)
         - servers.json (same format)
```

### Migration Strategy

**Backward Compatibility:**
1. Phase 1 users keep their setup - Scripts continue working
2. Web UI added as optional component - `docker-compose.webui.yml`
3. Gradual adoption - Users switch when ready
4. No breaking changes - All Phase 1 agents work in Phase 2

**Web UI Integration Points:**
- **Configuration** - Reads/writes `.env` files instead of interactive prompts
- **Orchestration** - Calls `docker compose up/down` via API instead of scripts
- **Monitoring** - Displays `docker compose logs` in web interface
- **Multi-Agent** - Manages multiple domain agents simultaneously

**Transition Path:**
```bash
# Phase 1: Current
```

## Risks & Mitigations

### Risk 1: Docker Installation Complexity

**Risk:** Users struggle to install Docker Desktop

**Mitigation:**
- Provide clear installation links in error messages
- Create video tutorial for Docker installation
- Offer pre-configured VM images as alternative (Phase 2)

### Risk 2: API Key Confusion

**Risk:** Users don't know where to get API keys

**Mitigation:**
- README includes direct links to API key pages
- setup.sh shows helpful links during prompts
- Common errors link to API key documentation

### Risk 3: Platform-Specific Issues

**Risk:** Scripts fail on certain OS versions or shells

**Mitigation:**
- Test on multiple OS versions (Ubuntu 22.04, macOS 13+, Windows 10/11)
- Use POSIX-compliant shell syntax
- Provide troubleshooting section in README

### Risk 4: Network Connectivity

**Risk:** Image pull fails in restricted networks

**Mitigation:**
- setup.sh continues even if pull fails (retries on start)
- Document manual image load process
- Consider offering image as direct download (Phase 2)

### Risk 5: Permission Issues

**Risk:** Linux users hit Docker permission errors

**Mitigation:**
- Detect permission errors and show solution
- Document usermod command clearly
- Consider rootless Docker documentation
